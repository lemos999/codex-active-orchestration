# Context Router

Name class/weight: `tiny`, `standard`, or `heavy`: coding, review, research, assembly, measurement, delegation, explanation, command. Choose smallest finishable path; load only required files/tools/skills/refs.

If intent is unclear, interview before implementation, research, broad inspection, or explanation. Ask 1 to 3 short questions deciding advice, design, implementation, comparison, verification, packaging, explanation target, or other. Skip polish until path is clear.

Broad codebase prompts like `Explain this codebase`, repo walkthroughs, or project overviews are not `micro` until narrowed. Ask audience/scope/depth: overview, architecture, module flow, onboarding, risk review. If full tour is explicit, route `macro`, inspect entrypoints/manifests/README, summarize, then deepen.

Treat wording as evidence, not keyword trigger. If follow-up still leaves deliverable, mode, target, or breadth unclear, ask the next smallest path question. After answer, route to smallest needed skill and return to `micro` once stable.

Accounting is routing. If invoked, `context-router` is used even when its file was not opened. Never report `none` or `0 skills` after naming a skill route. Separate opened skill files from used skills. Count only actually routed/invoked skills; similar tactics do not count as `output-slicing` or another skill.

Coding: repo shape and likely owners before broad dirs. Research: define question/evidence before sources. Long logs/diffs/search/JSON=>`output-slicing`. Volatile claims=>`source-fidelity`; token claims=>`baseline-observer`; delegation=>`subagent-gate`; graph=>`graph-locator-boundary`.

Ledger: class, intent, questions, loaded files, used skills, opened skill files, omitted context, next step.

Load repo-root `references/operating-philosophy.md` only for unclear routing.
